\section{Introduction}
This report presents the design, implementation, and evaluation of a distributed task queue system.
The project studies trade-offs in communication overhead, scalability, and operational complexity across two distributed designs:
\begin{enumerate}[label=(\alph*)]
    \item a microservice architecture that separates functionality into dedicated services, and
    \item a monolith-per-node architecture where each node colocates most system functionality.
\end{enumerate}
Both designs use gRPC-based RPC communication and Dockerized deployment.

\section{System Overview}

\subsection{System Design}
The system accepts client jobs, schedules work, executes tasks, and returns results.
In the microservice architecture, the distributed system includes six \emph{functional} containerized nodes:
\begin{enumerate}
    \item Gateway Service
    \item Job Service
    \item Queue Service
    \item Coordinator Service
    \item Worker Service
    \item Result Service
\end{enumerate}

\noindent
The project uses a separate load-generator client for benchmarking.
The load generator is evaluation tooling and is not counted as one of the six functional system nodes.

\subsection{Communication Model}
The system uses gRPC for client-service and service-service communication.
gRPC provides strongly typed interfaces through Protocol Buffers, compact binary serialization, and explicit service contracts.
This model makes service boundaries clear and supports reproducible latency and throughput measurements.

\subsection{Compared Designs}
\begin{itemize}
    \item \textbf{Design A (Microservices):} six dedicated functional services (Gateway, Job, Queue, Coordinator, Worker, Result).
    \item \textbf{Design B (Monolith-per-node):} six identical nodes, each colocating most functionality in-process while exposing equivalent external APIs.
\end{itemize}

\noindent
\textbf{API equivalence constraint:}
To ensure a fair comparison, both designs expose the same client-facing gRPC API
(\texttt{SubmitJob}, \texttt{GetJobStatus}, \texttt{GetJobResult}, \texttt{CancelJob}, and \texttt{ListJobs})
with equivalent request/response schemas and status semantics.
The experiments use identical load-generation and measurement logic across both designs.


\subsection{Metrics}
\begin{itemize}
    \item Throughput (requests/sec and jobs/sec)
    \item Latency (p50, p95, p99 per API)
    \item End-to-end job completion time (submit-to-done)
\end{itemize}

\subsection{Fairness Controls}
To keep the comparison fair, both designs use the same hardware, workload profiles, test durations, and measurement procedures.
The experiments also use identical client-side measurement logic and warm-up behavior across both designs.
